## O1.2 — Bad SHA as-is (key finding, BACKLOG candidate)

**Finding:** `golden/check-schema-safety.sh:33` uses SHA `4b825dc8669f8c0` (15 chars). The correct empty-tree SHA is `4b825dc642cb6eb9a060e54bf8d69288fbee4904` (40 chars). Both differ.

**Observed behavior:** On a 1-commit repo, `git diff HEAD~1..HEAD` fails (no parent) → `git diff 4b825dc8669f8c0..HEAD` also fails (invalid object, not a valid SHA) → `echo ""` fires → Branch C ("safe"). Outcome: 1-commit repos are always "safe" (no data-loss risk on first deploy — correct outcome, wrong mechanism).

**Port decision:** AS-IS per Luật chơi 1 (parity before improvement). The functional outcome (Branch C on first commit) is correct. BACKLOG improvement: fix to `4b825dc642cb6eb9a060e54bf8d69288fbee4904` (proper empty-tree SHA) — would make the empty-tree comparison actually work and close the gap for repo-with-1-commit-vs-empty-tree case.

**Cite:** `golden/check-schema-safety.sh:33`, ported at `src/checks/schema.rs` `get_diff()` Step 2.
